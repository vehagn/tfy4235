Number representation

Dealing with 1-byte integers (char) in the following:

Example:

The integer $213$ is, in base-10, decomposed into powers of ten,

$$213 = 200 + 10 + 3 = 2\cdot100 + 1\cdot10 + 3\cdot1$$

Similarly, the same number can be decomposed into powers of two,

$$213 = 128 + 64 + 16 + 4 + 1 = 1 \cdot 2^7 + 1 \cdot 2^6 + 0 \cdot 2^5 + 1 \cdot 2^4 + 0 \cdot 2^3 + 1 \cdot 2^2 + 0 \cdot 2^1 + 1 \cdot 2^0$$

Therefore, the number $213$ is, in binary representation,

$$213_2 = 11010101$$

In general, for any base, b

$$x_b = \sum_{i}a_i \cdot b^i, ~~a_i \in {0,..,b-1}$$

We need a way to check every bit of an integer.

Bitwise operators

We need two bitwise operators, the bitwise AND, and the left shift.

Bitwise AND, (&)

The bitwise AND operator does the bitwise AND operation on all the bits of an integer simultaneously, using the following rules

1 & 1 == 1
1 & 0 == 0
0 & 1 == 0
0 & 0 == 0

So, as an example,

213:      11010101
53:       00110101
213 & 53  00010101   // this is equal to 16+4+1 == 21

therefore, 

213 & 53 == 21

Bitwise left shift, (<<)

The left shift operator moves every bit in an integer one bit to the left, removing excess bit values, and appending zeros at the start,

213:      11010101
213 << 1: 10101010   // equal to 128 + 32 + 8 + 2 == 170

In base-10, every power-of-ten has the first digit as one, and the rest zeros. Similarly for binary numbers, a power-of-two has the first bit as one and the rest as zero.

00000001 == 1
00000010 == 2
00000100 == 4
00001000 == 8
etc.

We can create powers-of-two by using the left shift operator on the number 1. Combining this with the bitwise AND operator, we can check every bit in an integer.

int x = 213;

int bit = x & (1 << 3);    // here we check the 4th bit, 2^3 == 8
In [24]:
213 & (1<<3)
Out[24]:
0
In [25]:
213 & (1<<4)
Out[25]:
16

NB: Doesn't output only 1 or 0, 

213       11010101
8         00001000
213 & 8   00000000   // equal 0

213       11010101
16        00010000
213 & 16  00010000   // equal 16 !

To print only ones or zeros, we need to either negate twice (logical not, !), or compare to zero:

!!(213 & (1<<3))

This will give 1 (true) if the 4th bit of 213 is one, and 0 (false) if the 4th bit is zero.

In [22]:
213 & (1<<3) != 0
Out[22]:
False
In [23]:
213 & (1<<3) == 0
Out[23]:
True

Floating point (real) numbers

We cannot use bitwise operators on floating point numbers, and if we try to cast a float to an int, we will not get what we want.

float a = 1.3;
int b = (int)a;   //this will evaluate to b == 1

Instead, we need to trick the program into reading the float as an int. Two methods,

  1. (Ab)using pointers
  2. Using unions

Pointers

We use pointers, which points to a memory location. A float pointer tells the program that what's in the memory location it points to should be read as a float type. We can use the address operator (&, OBS!), to get the address of a float variable, and let an integer pointer point to this location:

float a = 1.3;

int * b = &a;

print("%d\n", &a);    // will print the same thing as
print("%d\n", b);     // this, the memory location of "a"

we use the dereference operator (*) to get what's in the memory location the pointer points to.

print("%d\n",*b);

this will print an integer, having the same bit representation as the bit representation of the floating point number "1.3". the value pointed to by "b" in our case can be read as an integer, and the bit representation can be read by the method for normal integers.

Alternative: using functions:

void print_float(int * in){
    print_int_as_binary(*in);
}

and call the print_float function as

float a = 1.3;
print_float(&a);

This approach is to do a programming error on purpose.

Unions

This is the proper way to have several representations of a variable.

typedef union my_union {
    float f;
    int i;
} union_t;

int main () {

    union_t a;

    a.f= 1.3;

    printf("%d\n",a.i);    // will print the corresponding integer representation of the float.

    return 0;
}

Use long int for doubles.

Problem 2, precision

One possible definition of floating point precision would be

The smallest possible number such that adding this number to 1 gives a result different from 1.

Start by 1.0, divide by 2, repeat

Extra, round-off errors

Floating point operations naturally lead to round-off errors. In addition, floating point operations are not associative, so the round off errors depends on the order in which one performs the operations.

The way, say, floating point addition works is that the mantissa of the number having the smallest exponent is shifted such that the exponents are the same, then the two mantissa are added by integer addition.

Therefore, if you add a very small number to a very large number, you will get significant round-off errors. This happens if you calculate the sum of many floats.

Solutions,

  1. Pairwise summation: Recursively divide the sum into two parts, then sum the two parts. That is, first sum the first and second element, the third and fourth element, etc. Then sum the first and second partial sum, the third and fourth, and so on. Relatively cheap.

  2. Kahan summation: keep track of the round off errors.

Pseudocode from wikipedia:

function pairwise(x[1…n])
    var sum;
    if n ≤ N                    // base case: naive summation for a sufficiently small array
      s = x[1]
      for i = 2 to n do
          s = s + x[i]
    else                        // divide and conquer: recursively sum two halves of the array
      m = floor(n / 2)
      s = pairwise(x[1…m]) + pairwise(x[m+1…n])
    endif

    return s

function KahanSum(input)
    var sum = 0.0
    var c = 0.0          //A running compensation for lost low-order bits.
    for i = 1 to input.length do
        y = input[i] - c    //So far, so good: c is zero.
        t = sum + y         //Alas, sum is big, y small, so low-order digits of y are lost.
        c = (t - sum) - y   //(t - sum) recovers the high-order part of y; subtracting y recovers -(low part of y)
        sum = t             //Algebraically, c should always be zero. Beware eagerly optimising compilers!
        //Next time around, the lost low part will be added to y in a fresh attempt.
    return sum

Kahan summation will give the same result no matter the order one does the summation, but is expensive.
